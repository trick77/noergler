Design note · noergler · multi-team rollout

One team today, ten tomorrow

How to serve ten teams without handing any single account the keys to every repository — and without ten review bots stampeding the Bitbucket and Jira servers they all share.

Assessment Target-state, not current-state Decision pending
Constraints this rests on

Serialising reviews is a deliberate choice to avoid overwhelming Bitbucket and Jira, not an implementation accident — so it stays. noergler itself is open to change, so “needs no code change” is not a tiebreaker. Every team must supply its own inference key and its own Bitbucket service account; Jira may share a read-only account. Together these settle the topology on a single instance — with a per-team webhook secret carrying the identity that attribution depends on.

Direction
One instance serving every team, resolving credentials per team with a fallback — and one mandatory, non-negotiable inference key per team.

Serialisation protects shared infrastructure, and protection of a shared resource has to be global to mean anything. Once credentials resolve per request rather than per process, a single instance stops being in tension with per-team isolation.

The credential resolution rules

Two mandatory, one shared — deliberately not a single policy for all three:

Per team, per credential
CredentialPolicyIf absent
Inference key Per team. Mandatory — this is what makes provider-side spend attributable. Fail closed. Refuse the review, say so visibly. Never silently borrow another team's budget.
Bitbucket Per team. Mandatory — a team's own service account is what makes repository access opt-in. Fail closed. No shared account to inherit, so nothing accumulates broad access by default.
Jira Per team when supplied. Fall back to the shared read-only account. Lowest risk of the three.

Validate the mandatory key at registration time, not at first review. A team that discovers its configuration is incomplete when a pull request goes unreviewed learns it in the worst possible place.

The argument for a single instance is sound

Bitbucket and Jira are shared infrastructure. A limit enforced per instance is not a limit on them — ten instances each politely running one review at a time still present ten concurrent review streams to a server that was only ever measured against one. Whatever headroom the current setup demonstrates does not survive being multiplied by ten.

That reasoning is correct, and it's the strongest argument on the table. The question it leaves open is whether a single process is the only way to get a single budget.

A caveat on what is protected today

The single review worker bounds how many pull requests are in flight. It does not bound how many requests reach Bitbucket, because each review fans out internally:

  • Per-review file fetches are unbounded. app/reviewer.py:402 runs asyncio.gather across every reviewable file in the diff, each calling fetch_file_content. A 120-file pull request issues 120 concurrent fetches.
  • Nothing caps the connection pool. The httpx.AsyncClient in app/bitbucket.py:31 sets a timeout but no limits=, so it inherits the default ceiling of 100 concurrent connections.

So today's protection is real but coarse: one large pull request can already burst to roughly a hundred concurrent requests. The mechanism that genuinely matches the intent is an explicit concurrency budget on the Bitbucket and Jira clients — which is worth adding regardless of which topology wins.

Team identity comes from the secret, not the payload

If attribution has to be certain, it cannot rest on a field inside the request body. The project key in a webhook payload is unauthenticated until the signature is checked — it says who the sender claims to be. A per-team webhook secret makes the claim provable: the team that signed it is the only team that could have.

The mechanism is already half-built. scripts/onboard_repo.py programs both the webhook URL and configuration.secret per repository, and reports drift when either differs from the desired state. Giving each team its own URL and its own secret is a change to that script's config file, not new machinery.

Verify in this order

  • The webhook path names the team, so the secret is selected before any parsing — this replaces the single global secret checked at app/main.py:239.
  • The signature verifies against that team's secret. Failure is rejection, not a fallback to another team's key.
  • The payload's project belongs to that team. Skipping this step is the security hole: a team holding a valid secret could otherwise sign a payload naming another team's repository, and have it reviewed with credentials that team never granted.

Then record the authenticated team, not the claimed one

Attribution should be stored from the identity that survived verification. The database has no team column today — pr_reviews is keyed UNIQUE (project_key, repo_slug, pr_id) — so per-team cost and volume are currently inferred from the project key rather than known. Adding the team to that table turns inference into an audit trail.

That unique key has a second consequence worth settling now: if two teams onboard the same repository, both webhooks fire and both reviews contend for the same row. Decide that a repository belongs to exactly one team and enforce it at registration, where the answer is cheap, rather than in the database, where it is a corruption report.

What the single instance obliges you to build

Not a long list, but two items are easy to miss because they are invisible until a second team arrives:

  • The mention trigger is a global username. app/main.py:276 builds the trigger from config.bitbucket.username. With every team on its own bot account this is no longer optional — each team has a different handle, and a global trigger would silently ignore every mention. Worth settling a naming convention at the same time, since the handle is what people type.
  • Module globals become per-tenant lookups. The clients in app/main.py:43 are assigned once at startup. Each becomes a per-tenant instance resolved from the registry, cached per team.
  • The queue needs fairness. One worker shared by ten teams means one very large pull request delays everyone. Round-robin across tenants rather than first-in-first-out.

The counterargument worth weighing

A single instance holds every team's credentials, so it avoids an all-seeing account partly by creating an all-seeing process — one whose compromise exposes ten teams' tokens at once, and whose bad deployment stops code review everywhere. A namespace holding ten teams' secrets also cannot reasonably be owned by any one of those teams; it needs a platform owner.

Dropping the Bitbucket fallback closes the sharper version of this. A shared Bitbucket account would have accumulated access from every team that declined to bring its own, converging on the broad-access bot the design set out to avoid — the fallback would have been the default in practice, whatever the intent. Requiring each team's own account means no single token reaches across teams: the process holds ten credentials, but each one is narrow, and compromise of the process is the only path to breadth.

The price is onboarding friction. A team cannot start until it has created a service account and granted it access, and that step now blocks rather than degrades. Better to surface it at registration, where scripts/onboard_repo.py already performs a read-permission check, than at the first unreviewed pull request.

Decisions that hold either way

Settled
DecisionVerdict
Inference key per team Confirmed, for spend attribution. Under per-team instances it's an environment value; under one instance it's a registry lookup.
Bitbucket opt-in via per-team accounts Endorsed. A team's bot only reaches repositories that granted it access, and scripts/onboard_repo.py already performs the read-permission check plus an idempotent webhook upsert per repository.
One shared Jira account Safe. The Jira client only calls fetch_ticket — read-only. Caveats: a single rate-limit bucket at ten times the traffic, and one rotation affecting everything at once.
Serialised reviews Keep the intent, change the mechanism. Move the limit from “one review at a time” to an explicit budget on outbound Bitbucket and Jira calls, so backend load stays bounded while a single team's review can still parallelise internally.

Still open

What is the actual backend budget?

The whole design rests on a number nobody has measured: how many concurrent requests Bitbucket and Jira tolerate comfortably. One review at a time was a safe guess for one team. Before ten teams arrive, that guess should become a measurement — otherwise the budget is set by superstition, and either throttles reviews needlessly or fails to protect anything.

Head-of-line blocking between teams

One queue shared by ten teams means nine wait behind the tenth's four-thousand-line pull request unless fairness is explicit. This is the cost of the single instance and the thing most likely to generate complaints in month one. Round-robin across tenants, and measure queue wait per team from day one rather than after the first escalation.

Who owns the namespace?

A single instance holding ten teams' credentials cannot live in a team-owned namespace. It needs a platform owner with a rotation process, an on-call route, and an upgrade cadence that ten teams can live with. This is an organisational decision, not a technical one, and it is the most common way shared-service rollouts stall.

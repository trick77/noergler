## How to read the diff

Each file below includes the **full content of the new (post-change) version** and a **minimal diff**.
In the diff:
- Lines starting with `-` are **removed** — this code no longer exists after the change.
- Lines starting with `+` are **added** — this is the new code introduced by the change.
- The full file content shows the final state and is the source of truth.

Only review the **added (`+`) lines**. Do not flag issues in removed (`-`) lines — that code is deleted.
Do not assume that removed code still exists elsewhere in the codebase unless you can see it in the full file content provided.
A rename shown as `-oldName` / `+newName` means only `newName` exists now.

## What to look for

**Bugs and logic errors** — off-by-one errors, incorrect conditions, null/undefined handling, race conditions, resource leaks, type mismatches, missing edge cases.

**Security vulnerabilities** — injection flaws (SQL, command, XSS), authentication/authorization issues, sensitive data exposure, insecure deserialization, path traversal.

**Performance problems** — unnecessary allocations in hot paths, O(n^2) or worse where avoidable, missing indexes, unbounded queries, blocking calls in async contexts.

**Code quality** — duplicated logic, missing error handling, broken error propagation, incomplete refactors, dead code introduced by the change, test coverage gaps for new logic.

## What NOT to flag

- **Code comments, docstrings, and string literals** — these are not executable code. Do not flag patterns mentioned in comments as if they were actual code. A comment describing a workaround is not the same as the workaround being present in code. However, DO flag comments that are clearly outdated or contradict the surrounding code, as misleading comments should be corrected.
- **Best-practice additions not in the code** — do not suggest adding properties, decorators, configurations, or patterns just because they are considered a best practice or modern convention. If the code works without `standalone: true`, a particular annotation, or a specific config flag, do not flag its absence. Only flag what is actually broken or violates the project guidelines provided above.
- **Removed code** — lines starting with `-` in the diff are deleted. Do not treat them as current code, do not warn about inconsistencies between removed and added code (that is the *point* of the change), and do not speculate that removed code may still exist elsewhere.
- **Speculation about unseen code** — the full file content provided is the complete, authoritative post-change state. If a name, variable, or method does not appear in the full file content, it does not exist. Do not speculate that removed code "may still be referenced elsewhere", that callers "might not be updated", or that consistency problems "could exist" outside the provided files. Only flag issues you can directly observe in the code provided.
- **Correct refactors** — if a rename (`-oldName` / `+newName`) is consistently applied throughout the full file content (the old name no longer appears, the new name is used everywhere), this is a completed refactor, not a bug. Do not flag it.

## Confidence scoring

Assign each finding a confidence score from 0 to 100 indicating how certain you are that the issue is real and actionable. Only report issues with confidence >= 80. This filters out speculative or stylistic complaints.

## Severity levels

Based on the confidence score, assign a severity:
- **"critical"** (confidence 90-100): Critical issues — bugs that will manifest at runtime, security vulnerabilities, data loss risks.
- **"warning"** (confidence 80-89): Important issues — likely bugs, error handling gaps, performance problems that matter at scale.

Do NOT report stylistic issues. Only report "critical" and "warning" findings — problems that a senior engineer would insist on fixing before merging.

## Output format

Respond with a JSON array. Each item must have:
- "file": the file path relative to the repo root (e.g. "src/main/java/Foo.java"), without any "a/" or "b/" diff prefix

- "severity": "critical" or "warning"
- "confidence": integer from 80 to 100
- "comment": a concise description of the problem. Use markdown: wrap code references in backticks (`like this`). Do not include fix code in the comment.
- "suggestion": (optional) the corrected code snippet as raw code (no markdown fencing). Omit this field if no concrete fix applies.
